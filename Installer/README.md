# Installer

## Inventory

- Built executables: `TrayApp.exe` and `TrayService.exe`.
- Release x64 build outputs used by the installer: `x64\Release\TrayApp.exe` and `x64\Release\TrayService.exe`.
- Files installed into `Program Files\ZIOVPO Antivirus\`: `TrayApp.exe` and `TrayService.exe`.
- External DLLs shipped with the app: none. Both binaries currently depend only on Windows system DLLs plus the MSVC/UCRT runtime installed separately.
- TrayApp external resources: none. Icons, strings, and UI resources are compiled into the executable via `Antivirus.rc`.
- External config files: none. No `.json`, `.ini`, `.config`, `.xml`, or similar runtime config files are loaded from the install directory.
- Default antivirus database files shipped as files: none. `default.avdb` is generated by the service on first start under `ProgramData\Antivirus\Databases\`.
- ProgramData runtime layout: `ProgramData\Antivirus\Databases\` stores `main.avdb`, `backup.avdb`, `default.avdb`, `incoming.avdb`, and `database.tmp` when the service creates or updates them.
- Third-party frameworks used: no Qt, no Windows App SDK, no .NET runtime.
- CRT mode in the actual `Release|x64` VCXPROJ outputs: dynamic MSVC runtime. `dumpbin /dependents` shows `MSVCP140.dll`, `VCRUNTIME140.dll`, `VCRUNTIME140_1.dll`, and UCRT API-set imports, so the bundle installs the official VC++ Redistributable x64.
- Windows service name from code: `AntivirusService`.
- Service display name used by the installer: `ZIOVPO Antivirus Service`.

## What This Installer Does

- Installs the app per-machine into `C:\Program Files\ZIOVPO Antivirus\`.
- Registers `AntivirusService` with `StartType=auto`, account `LocalSystem`, binary path pointing to installed `TrayService.exe`.
- Starts the service after install.
- Stops and removes the service via SCM on uninstall and upgrade using WiX `ServiceControl`.
- Removes the install directory on uninstall.
- Removes app-owned database files from `ProgramData\Antivirus\Databases\` on uninstall.
- Installs the official VC++ Redistributable x64 through the bundle because the shipped binaries use the dynamic MSVC runtime.

## What Is Not Required

- Qt is not used. There are no Qt libraries, project references, or runtime assets.
- Windows App SDK is not used. The code is plain Win32/C++ with native Windows libraries.
- .NET is not used. There are no managed assemblies or .NET runtime dependencies in the app itself.

## Local Build

1. Build the native binaries:

```powershell
& "C:\Program Files\Microsoft Visual Studio\18\Insiders\MSBuild\Current\Bin\amd64\MSBuild.exe" Antivirus.slnx /restore /t:Build /p:Configuration=Release /p:Platform=x64
```

2. Build the MSI:

```powershell
& "C:\Program Files\Microsoft Visual Studio\18\Insiders\MSBuild\Current\Bin\amd64\MSBuild.exe" Installer\TrayInstaller.wixproj /restore /t:Build /p:Configuration=Release /p:Platform=x64
```

3. Build the bundle:

```powershell
& "C:\Program Files\Microsoft Visual Studio\18\Insiders\MSBuild\Current\Bin\amd64\MSBuild.exe" Installer\TrayBundle.wixproj /restore /t:Build /p:Configuration=Release /p:Platform=x64
```

If you need a release-specific MSI version for upgrade testing, add `/p:InstallerVersion=1.0.123`.

`collect-artifacts.ps1` runs automatically from the WiX projects. It clears the staging folder, copies the fresh `x64\Release\TrayApp.exe` and `x64\Release\TrayService.exe`, prints the staging timestamp and the source path of `TrayService.exe`, and optionally downloads `VC_redist.x64.exe` for the bundle build.

## CI Build

GitHub Actions builds the installer on `windows-latest` without relying on a preinstalled WiX toolset:

- installs .NET SDK explicitly;
- installs WiX Toolset `5.0.2` explicitly;
- restores WiX SDK and extension packages from the `.wixproj` files;
- builds `Antivirus.slnx` in `Release|x64`;
- runs `TrayService.exe --av-self-test`;
- builds `Installer\TrayInstaller.wixproj`;
- builds `Installer\TrayBundle.wixproj`;
- verifies that `TrayApp.exe`, `TrayService.exe`, the MSI, and the bundle exist;
- verifies that staged `TrayService.exe` matches the freshly built `x64\Release\TrayService.exe`;
- publishes separate GitHub Actions artifacts for raw app binaries and installer outputs.

Published artifacts:

- `ziovpo-antivirus-app-x64`: `TrayApp.exe`, `TrayService.exe`
- `ziovpo-antivirus-installer-x64`: `ZIOVPOAntivirus.msi`, `ZIOVPOAntivirusSetup.exe`, `README.md`

## Install

- MSI:

```powershell
msiexec /i Installer\bin\Release\x64\ZIOVPOAntivirus.msi
```

- Bundle:

```powershell
Installer\bin\Release\x64\ZIOVPOAntivirusSetup.exe
```

## Uninstall

- MSI:

```powershell
msiexec /x Installer\bin\Release\x64\ZIOVPOAntivirus.msi
```

- Bundle:

```powershell
Installer\bin\Release\x64\ZIOVPOAntivirusSetup.exe /uninstall
```

## Verify The Service

- Check service registration:

```powershell
Get-CimInstance Win32_Service -Filter "Name='AntivirusService'" | Select-Object Name, DisplayName, StartMode, State, PathName
```

- Check service security descriptor:

```powershell
sc.exe sdshow AntivirusService
```

- Run the full elevated verification:

```powershell
powershell -ExecutionPolicy Bypass -File Installer\scripts\verify-install.ps1 -InstallerPath Installer\bin\Release\x64\ZIOVPOAntivirusSetup.exe
```

The script verifies:

- install of MSI or bundle;
- presence of `TrayApp.exe` and `TrayService.exe`;
- creation of `default.avdb`;
- SCM registration of `AntivirusService`;
- `StartMode=Auto`;
- `sc.exe sdshow AntivirusService`;
- absence of DENY ACE for `BUILTIN\Users` and `BUILTIN\Administrators`;
- ability to stop and start the service through SCM;
- uninstall without `Service could not be stopped`;
- removal of the SCM service entry and install directory.

## Old Broken DACL Repair

If a historical installation left a broken service DACL that prevents normal SCM stop/delete, repair it first:

```powershell
sc.exe sdshow AntivirusService
sc.exe sdset AntivirusService D:(A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;SY)(A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;BA)
sc.exe stop AntivirusService
sc.exe delete AntivirusService
```

If an older experimental build ever registered the service under `TrayService`, use the same commands with `TrayService`.
